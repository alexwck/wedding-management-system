# Data Model: Speed Up Playwright E2E Suite

**Branch**: `014-e2e-speedup` | **Date**: 2026-06-06

This file is Phase 1 output for the `/speckit-plan` workflow. There is no domain
data model for this feature (no Supabase schema changes), but the test-infrastructure
artifacts have a small entity model worth pinning down.

## Entities

### Playwright Project

A named browser configuration defined in `playwright.config.ts`. Each project has
its own `use` (devices, baseURL, storageState), `dependencies` (projects that must
run before this one), and `testMatch` (which spec files belong to it).

| Attribute | Type | Notes |
|---|---|---|
| `name` | string | `setup`, `chromium`, or `Mobile Chrome` for this feature |
| `use.storageState` | path \| empty object | file path consumed by Playwright when initializing the test context |
| `dependencies` | string[] | project names; Playwright runs them first |
| `testMatch` | glob[] | empty for the active browser projects; `/.*\.setup\.ts/` for `setup` |

State: projects are static; declared once in config. No lifecycle.

### StorageState

A JSON file on disk holding cookies + localStorage for one logged-in user.
Generated by `tests/e2e/auth.setup.ts` via `context.storageState({ path })`.
Consumed by browser projects via `use.storageState`.

| Attribute | Type | Notes |
|---|---|---|
| `path` | string | `playwright/.auth/admin.json` or `playwright/.auth/couple.json` |
| `cookies` | array | Supabase session cookies (`sb-{project}-auth-token` and friends) |
| `origins` | array | only present if the user has localStorage on the origin |

State: rewritten by the setup project on every Playwright run. Never committed
(gitignored). Stale files cause confusing failures - the implementer should `rm
playwright/.auth/*.json` if a test fails with "missing session" after a DB reseed.

### Setup Project

A Playwright project whose `testMatch` excludes real specs and only runs the
authentication flow. Has no `storageState` of its own; writes state for dependents.

| Attribute | Type | Notes |
|---|---|---|
| `name` | string | always `setup` for this feature |
| `testMatch` | glob[] | `/.*\.setup\.ts/` so it only runs `auth.setup.ts` |
| `dependencies` | string[] | empty - this is the leaf |

State: Playwright invokes it once at the start of the run when listed as a
dependency. No per-spec lifecycle.

## Relationships

```
+-------------+  writes   +-----------------------+
| setup       | --------> | playwright/.auth/*.json|
+-------------+           +-----------------------+
                                      ^
                                      | consumed via use.storageState
                                      |
                          +-----------+-----------+
                          |                       |
                          v                       v
                    +-----------+          +----------------+
                    | chromium  |          | Mobile Chrome  |
                    +-----------+          +----------------+
```

## Validation rules

- A spec file that needs the non-default role storageState must call
  `test.use({ storageState: "playwright/.auth/{role}.json" })` at the top of the
  file (outside any `test.describe`).
- A spec file that needs a fresh context (login flow, logout, invalid login) must
  call `test.use({ storageState: { cookies: [], origins: [] } })`.
- A spec file that does not call `test.use` for storageState inherits the
  *project's* default. This is the dominant case (35 of 37 specs).
- A spec file may override `test.use` per-`test.describe` block if it needs
  different storageState for different flows (used by `admin-lock.spec.ts`).

## Out of scope

- No changes to the Supabase schema or seed data.
- No changes to application code (`src/`), unit tests, or component tests.
- No changes to the GitHub Actions workflow YAML.
