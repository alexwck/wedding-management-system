# Implementation Plan: Admin Lock, Floor Plan Polish & RSVP Redesign

**Branch**: `010-ux-polish-admin-rsvp` | **Date**: 2026-04-25 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/010-ux-polish-admin-rsvp/spec.md`

## Summary

7-part feature covering admin wedding lock (freeze all edits including admin), floor plan catalog collision blocking, save UX with OOB prevention, RSVP single-page redesign (merge landing + RSVP), editable couple name, template image caching fix, and undo/redo audit. Implementation is layered: database migration first, then server-side enforcement, then UI changes top-down from admin to couple to guest-facing.

## Technical Context

**Language/Version**: TypeScript (strict mode) with Next.js 16 (App Router) + React 19
**Primary Dependencies**: react-konva, konva, shadcn/ui (Nova theme), react-hook-form, zod, exceljs, cmdk
**Storage**: Supabase PostgreSQL with Row-Level Security
**Testing**: Vitest + React Testing Library (unit), Playwright (E2E, workers=1)
**Target Platform**: Web (desktop + mobile Chrome)
**Project Type**: Full-stack web application (Next.js App Router)
**Performance Goals**: Page loads <2s, undo/redo <200ms, auto-save within 5s of idle
**Constraints**: Glassmorphism design system (.glass-panel), mobile parity (onClick + onTap), adminClient bypasses RLS so lock checks must be server-action level
**Scale/Scope**: Dev stage, no backward compatibility, single admin, multiple couples

## Implementation Phases

### Phase A — Database & Server-Side Lock Infrastructure

Foundation layer. Everything else depends on this.

**Migration**: Add `is_locked BOOLEAN NOT NULL DEFAULT false` to `weddings` table.

**Server-side lock check**: Add `verifyWeddingNotLocked(weddingId)` helper to `src/lib/auth-guards.ts`. This helper:
- Fetches `is_locked` from weddings table via adminClient
- Throws/returns error if locked
- Called by every mutation server action between access verification and the actual mutation

**Mutation actions to guard** (lock check insertion point after auth+access verification, before adminClient mutation):
- `admin.ts`: `updateWeddingDetails` (~L388), `updateWeddingDate` (~L452), `updateWeddingTimezone` (~L486), `updateTemplateFocalPoint` (~L524), and new `toggleWeddingLock`, `updateCoupleName`
- `rsvp.ts`: `submitRSVP` (~L46, after wedding lookup), `updateRsvpStatus` (~L102)
- `floor-plan.ts`: `saveFloorPlan` (~L85, before upsert)
- `upload.ts`: `uploadTemplateImage` (~L33, before storage upload)

**New server actions**:
- `toggleWeddingLock(weddingId)`: Flips `is_locked` on weddings row. Admin-only. The ONLY mutation allowed on a locked wedding (it unlocks).
- `updateCoupleName(weddingId, coupleName)`: Updates `weddings.couple_name`. Guarded by lock check.

**RLS consideration**: Since all mutations use `adminClient` (bypasses RLS), lock enforcement must be in server actions, not RLS. The RSVP INSERT policy (public, no auth) also bypasses normal checks — the server action `submitRSVP` must check `is_locked` before allowing the insert.

### Phase B — Lock UI (Admin + Couple + Guest)

**Admin wedding detail page** (`admin/weddings/[id]/page.tsx`):
- Add `<LockToggle>` in the top bar (line 34 area), next to couple name
- Lock toggle: prominent button showing current state, calls `toggleWeddingLock`
- When locked: all edit sections show "Locked" badge, save buttons disabled

**Couple dashboard** (`dashboard/page.tsx`):
- Fetch `is_locked` in wedding data
- When locked: show banner "This wedding has been locked by admin. Contact admin for changes."
- Disable all edit controls: date picker, venue editor, template upload
- Couple name field: read-only when locked

**Public RSVP (guest-facing)**:
- On the merged single page (Phase D), show "RSVP is now closed" when locked
- The `submitRSVP` server action enforces this server-side

### Phase C — Floor Plan Polish (Catalog, Save UX, Undo/Redo)

**Catalog collision blocking** (`item-catalog.tsx`):
- New function `canPlaceItem(itemType, sizeVariant, existingItems, canvasWidth, canvasHeight)`: runs the existing spiral placement algorithm without actually placing the item, returns boolean
- Called on every canvas state change (items added/removed/moved, canvas resized)
- Returns a map: `{ "round_table_3ft": true, "round_table_4ft": true, "round_table_5ft": false, ... }`
- Item catalog reads this map: disabled items get `opacity-50 pointer-events-none` + tooltip "No space available"
- Performance: only compute when catalog is open/visible (avoid unnecessary computation)

**Save UX overhaul** (`use-auto-save.ts` + `floor-plan-canvas.tsx`):
- Clear status labels: "Unsaved changes" → "Saving..." → "All changes saved (timestamp)" → "Save failed — try again"
- OOB guard: before save, iterate items and call `isItemOutOfBounds(item, width, height)`. If any OOB: set status to "blocked" with count, skip save. Show "N item(s) outside canvas — move within bounds to save"
- Auto-save resumes automatically when all items are in bounds
- Remove ambiguous "Save Now" button label — use "Save" or just auto-save indicator
- Lock check: auto-save sets `enabled=false` when wedding is locked

**Undo/redo audit** (`use-undo-redo.ts`):
- Verify all actions tracked: catalog placement, delete, drag, rotate/resize, canvas dims, chair count, guest assign/unassign, label edit
- Verify one entry per gesture (not per intermediate state)
- Verify full state restoration (items + dims + assignments + unassigned guests)
- Fix any gaps found

### Phase D — RSVP Single-Page Redesign

**Merge landing page + RSVP form** into `/w/[slug]/page.tsx`:
- Fetch wedding data including all fields needed for both hero and RSVP
- Sections (top to bottom):
  1. **Hero section**: Full-screen template image (or gradient fallback), couple name, date with timezone, venue summary, RSVP CTA button
  2. **Venue section**: Venue name, address, welcome message, OSM map, navigation buttons (if venue data exists)
  3. **RSVP section**: RSVP form with anchor ID for scroll target, inline success state, "RSVP closed" when locked
- Remove 404 for missing template image — show gradient fallback hero
- RSVP CTA: `<a href="#rsvp">` with `scroll-behavior: smooth` on the page
- After RSVP submit: show inline success message, don't navigate away
- Delete `src/app/(public)/w/[slug]/rsvp/` directory entirely

**Fallback hero design** (no template image):
- Glassmorphism panel with gradient background
- Couple name + date centered
- RSVP CTA below

### Phase E — Polish (Couple Name, Template Fix, Rename)

**Editable couple name** (`editable-couple-name.tsx` + `admin.ts`):
- Inline editable component: click to edit, blur or Enter to save, Escape to cancel
- Positioned above date picker on admin detail page and couple dashboard
- Calls `updateCoupleName` server action
- Zod validation: non-empty string, max length (100 chars reasonable default)
- Read-only when locked

**Template image caching fix** (`upload.ts`):
- After upload, append cache-bust query param to public URL: `${publicUrl}?t=${Date.now()}`
- Return this URL as `template_image_url` so all consumers see fresh image
- Template preview component refreshes after upload

**Template preview rename** (`template-preview.tsx`):
- Rename button text from "Preview" to "Adjust Crop"
- Dialog title: "Adjust Image Crop" instead of "Preview"
